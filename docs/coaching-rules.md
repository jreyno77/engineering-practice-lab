# Coaching Rules

These rules guide the training loop and should be followed for future assignments, reviews, and progress updates.

## Public Repository Safety

This repository is public. Training content must stay generic and must not expose private work details.

Do not include:

- employer, client, or team-specific details
- internal project names
- private system names
- proprietary workflows
- private data, examples, URLs, screenshots, headers, credentials, or logs
- domain-specific details that could connect the training repo back to private work

Use generic examples instead. If a real work scenario inspires an exercise, rewrite it as a neutral training scenario with no identifying details.

## Daily Assignment Learning Direction

Daily assignments should include a small learning direction section before the task.

Each assignment should usually include one or more of:

- a short reading topic
- a documentation page to look up
- a book chapter or concept to review
- a YouTube/video topic to search for
- a small example to study before starting

The learning direction should point the user toward the concept without giving away the whole answer.

## Training Review Closure

For training and baseline submissions, the goal is to preserve the learning history.

The normal loop is:

1. Assignment is given.
2. User submits the work.
3. Review comments and score are added.
4. User responds with understanding or disagreement.
5. Score and summary are recorded in the repo.
6. Move forward unless there is disagreement, confusion, or production-intended project code that needs fixes.

Do not rewrite the submitted work into the reviewer’s ideal answer just to make the record look perfect. The original attempt, feedback, and understanding are the useful evidence.

## Practice Variety

Assignments should include concepts the user does not necessarily think about every day at work.

Good areas to rotate through:

- type-safe boundary parsing
- runtime validation
- API design
- data modeling
- error modeling
- testing strategy
- refactoring safety
- performance reasoning
- architecture boundaries
- product shaping
- technical writing
- disagreement and tradeoff communication
